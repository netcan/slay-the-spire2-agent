## Context

当前仓库的 `Sts2Mod.StateBridge` 已能在真实 STS2 引用下构建 DLL，并自动生成 `mod_manifest.json`，但 `.pck` 仍需手工准备。根据已验证的 STS2 mod loader 行为，loader 会扫描游戏目录下的 `mods/`，以 `.pck` 作为资源包入口，并在同目录发现同名 `.dll` 时加载程序集。

因此，这次变更的核心不是新增 runtime 协议，而是把已有 bridge 产物封装为稳定、可复用的安装包输出。设计上需要兼顾两点：一是尽量复用现有 `mod_manifest.json` 生成逻辑，二是避免引入沉重的人工编辑器步骤，让命令行或脚本即可完成打包。

## Goals / Non-Goals

**Goals:**
- 让仓库能够自动生成 `Sts2Mod.StateBridge.pck`。
- 保证 `.pck` 与 DLL 一起输出到稳定目录，便于复制安装。
- 提供最小验证方式，确认 pack 内容与目录结构正确。
- 为后续一键安装脚本或发布流程预留明确入口。

**Non-Goals:**
- 本次不实现完整的发布流水线或版本发布到外部平台。
- 本次不处理游戏目录自动拷贝、覆盖确认或卸载逻辑。
- 本次不改变 bridge runtime 协议、端口或动作执行行为。

## Decisions

### 1. 采用仓库内最小打包资源而不是依赖手工 Godot 工程
在仓库内提供最小打包输入，仅包含 `mod_manifest.json` 及打包所需的基础资源目录，由脚本或命令行工具生成 `.pck`。这样可以避免每次打开 Godot 编辑器手工导出，也减少环境差异。

备选方案：
- 继续手工用 Godot 编辑器导出：放弃，因为流程不可重复且不适合自动化。
- 直接提交一个预编译 `.pck`：放弃，因为产物容易过期且不可追溯。

### 2. 复用现有 `BridgeOutputDir` 作为统一产物目录
`.pck`、`Sts2Mod.StateBridge.dll`、`mod_manifest.json` 应输出到同一目录，保持 STS2 loader 所需的邻接关系，也便于后续脚本做整目录复制。

备选方案：
- 把 `.pck` 单独输出到另一个目录：放弃，因为会增加安装与验证复杂度。

### 3. 打包流程先追求“可验证可安装”，不追求复杂资源管理
首版 `.pck` 只需要保证包含 `mod_manifest.json` 这类 loader 必需资源，并能与 DLL 组成最小可安装 mod。若后续需要更多 Godot 资源，再在同一打包框架上扩展。

备选方案：
- 一开始引入完整资源项目模板：放弃，因为当前 bridge 需求还很小，维护成本高。

### 4. 验证以脚本检查和目录断言为主
打包成功后，通过脚本检查 `.pck` 是否存在、DLL 是否同目录、manifest 是否已参与打包或已在输出目录可见，并在文档中给出安装位置。这样比依赖人工肉眼检查更稳定，也适合后续 CI。

## Risks / Trade-offs

- [本机缺少可用的 `.pck` 打包工具链] → 先明确依赖项与命令入口，并在脚本中给出可诊断错误。
- [STS2 loader 对 pack 内容有额外隐式要求] → 首版以最小 pack 验证，必要时补充最小资源模板。
- [打包结果与 DLL 不同步] → 将 `.pck` 打包挂到现有构建输出目录，减少人工搬运。
- [后续自动安装脚本耦合过早] → 先稳定产物结构，安装逻辑留到后续 change。

## Migration Plan

1. 选定 `.pck` 打包工具链并在仓库内固化入口。
2. 把 `mod_manifest.json` 与最小资源布局纳入打包输入。
3. 构建后生成 `.pck` 到统一输出目录，并补充脚本校验。
4. 更新本地开发文档，说明如何构建、验证和安装。

回滚策略：
- 若自动打包暂时不稳定，保留现有 DLL + manifest 构建流程，仍可手工准备 `.pck` 继续联调。

## Open Questions

- 当前最合适的 `.pck` 生成方式是 Godot 命令行导出、专用 pack 工具，还是仓库内自定义最小打包器？
- 是否需要把 `.pck` 内的 `mod_manifest.json` 与输出目录中的副本同时保留，以兼容不同排障场景？
