## 为什么

目前已经可以从 Slay the Spire 2 的 mod 方向确认，游戏内部状态读取和操作驱动具备实现基础，但项目还没有定义一套稳定的 agent 接入契约，使外部策略程序能够以结构化方式读取局面并提交合法动作。现在先把 STS2 agent 的基础规范定下来，可以避免后续大模型方案退化成脆弱的界面脚本，也能为启发式策略、LLM 策略和回放调试提供统一底座。

## 变更内容

- 定义本地 `game-bridge` 契约，用于暴露结构化观察、合法动作集合、动作执行结果和对局生命周期控制能力。
- 新增 `autoplay-orchestrator` 能力，用统一流程拉取状态、调用策略、逐步执行动作，并记录完整决策轨迹。
- 统一 STS2 agent 所需的 JSON 数据模型，覆盖战斗回合、目标选择、奖励选择、地图分支和终局结果等核心决策窗口。
- 补充自动打牌的初始安全约束，包括阶段感知的动作校验、幂等/防过期提交、人工停止与中断处理机制。

## Capabilities

### New Capabilities
- `game-bridge`: 通过稳定的本地桥接契约暴露 Slay the Spire 2 的结构化状态、合法动作和对局生命周期控制能力。
- `autoplay-orchestrator`: 基于 `game-bridge` 驱动外部策略执行自动打牌流程，并输出可回放、可调试的决策轨迹。

### Modified Capabilities

None.

## Impact

- 为游戏侧 mod 和外部 agent 进程建立明确的接口边界。
- 为观察数据、动作模型、会话状态和 trace 输出建立统一规范。
- 为后续 C# mod、agent 运行时、测试夹具和联调流程提供实现依据。
