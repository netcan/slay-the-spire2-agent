## Context

bridge 已经可以从真实 STS2 运行时读取手牌、动作与窗口状态，但此前的卡牌身份生成更多依赖名称、费用、升级状态或不稳定的运行时字段。对于牌堆构成简单、重复基础牌很多的战斗场景，这种策略会让两张同名牌落到同一个标识上，最终影响 `snapshot`、`actions` 与 `apply` 的一致性。

问题不仅是显示层面重复，更关键的是执行链路不安全：如果 `play_card` 只按名称匹配，agent 即使拿到了某个 action，也无法确信实际被打出的就是那一张实例牌。因此需要把“卡牌实例身份”提升为 bridge 对外协议的一部分。

## Goals / Non-Goals

**Goals:**
- 为 live `snapshot.player.hand` 中的每张牌生成稳定、可区分的实例级 `card_id`。
- 让 `actions[].params.card_id`、`action_id` 与 `apply play_card` 的执行目标全部围绕同一个实例级标识对齐。
- 在重复手牌场景下保持 `decision_id`、动作集合与执行结果稳定可预期。
- 通过测试和真实游戏日志验证重复 `打击`、`防御` 等案例。

**Non-Goals:**
- 不尝试为所有游戏对象建立永久跨战斗、跨房间的全局唯一 id。
- 不在本次变更中扩展新的动作类型或窗口类型。
- 不改变用户向文本字段的展示策略，除非它直接影响身份稳定性。

## Decisions

1. `card_id` 采用实例特征与手牌位置联合生成
   - 优先读取可用的运行时实例字段；若字段不稳定或无法区分重复牌，则强制纳入 `handIndex` 等位置特征。
   - 这样即使两张牌名称、费用、升级状态完全一致，也能在同一决策窗口中区分开。
   - 备选方案是仅信任游戏内部某个实例字段，但真实运行时已证明这不足以覆盖所有重复牌场景。

2. `play_card` action 与手牌实例严格对齐
   - `actions[].params.card_id` 必须直接引用 `snapshot.player.hand` 中的实例级 `card_id`。
   - `action_id` 的生成也纳入 `card_id`，确保同名牌对应不同 action。
   - 这样外部 agent 不必再通过 label 猜测目标牌。

3. `apply play_card` 改为按实例级 `card_id` 匹配
   - 执行阶段先根据 `card_id` 在当前 live 手牌中寻找具体实例，再走实际出牌逻辑。
   - 若找不到对应实例，则拒绝请求并返回明确错误，而不是回退到按名称模糊匹配。
   - 这样可以把错误暴露给 agent，而不是 silently 误打牌。

4. 身份稳定性优先于视觉简洁性
   - `name`、`label` 仍保持用户向文本；稳定身份信息放在 `card_id`、`action_id` 与 `params`。
   - 这能兼顾 agent 的结构化调用和玩家可读性。

## Risks / Trade-offs

- [手牌位置参与标识后，重排会导致 `card_id` 变化] -> 接受该行为，并通过 `decision_id` / `state_version` 明确表达状态已推进。
- [不同运行时版本字段差异较大] -> 保留多级 fallback，优先保证“可区分”，其次才是“完全依赖内部实例字段”。
- [执行阶段比读取阶段更严格，可能暴露更多 stale action] -> 这是预期行为，明确失败优于误执行。
